# *ARRgh v0.1.1 — Sync Progress Visibility + Cover Resilience

**Released**: 2026-05-28

---

## What's new

### Sync progress log

Adding a title to the library (or manually re-syncing) now shows live progress instead of a static "Building…" spinner.

- **Library card** — spinning overlay updates in real time: "Searching MangaDex…", "Matched in Comick: …", "Synced 383 chapters from comick", etc. Card is now always clickable — you can navigate to the title while it builds.
- **Title detail page** — "Last Sync" panel shows the full log of every step. Collapsed by default; click to expand. Clears at the start of each new sync.
- **Synopsis** — also collapsed by default to reduce visual noise.

Implementation: `sync_log` table (migration `0036`), append-only per sync run, polled every 2 s. See ADR 0022.

### New API endpoint

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/titles/:id/sync-log` | Returns log entries for a title in insertion order |

### Cover thumbnail recovery after rebuild

Covers no longer permanently 404 after a Docker image rebuild.

- `GET /api/media/cover/:id` now falls back to the cached MangaUpdates CDN URL (stored in `title_meta`) when the local file is missing. The CDN URL is restored to `titles.cover_url` so subsequent requests bypass the lookup.
- Previously, a missing local file would null out `cover_url` and leave the thumbnail permanently broken until the title was re-added.

### Download reliability

- **Downloader** now fails fast with an error (instead of writing an empty `.cbz`) when a source returns 0 pages for a chapter.
- **Sync FK race** — if a title is deleted between the `POST /titles/:id/sync` 202 response and the background spawn executing, the spawn exits cleanly instead of crashing with a FK constraint error.

---

## Bug fixes

- **MangaUpdates `latest_releases` 502** — `releases/search` could return `null` for `series_id`; now handled as `None` without an error.
- **Relative `download_dir` override** — `server_settings.download_dir` is now ignored if it is a relative path (e.g. `./downloads`). A relative path stored here caused covers to land in the container filesystem layer (not the volume), disappearing on every image rebuild. The env var `DOWNLOAD_DIR` is used instead and a warning is logged.

---

## Upgrading

```bash
docker compose pull
docker compose up -d
```

Migration `0036_sync_log` runs automatically on startup. No breaking changes.

**Fix existing broken thumbnails**: covers that were nulled out by the old behaviour will self-heal on first page load — `serve_cover` now redirects to the MangaUpdates CDN URL automatically.
